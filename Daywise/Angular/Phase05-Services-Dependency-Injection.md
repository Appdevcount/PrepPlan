# Phase 5: Services & Dependency Injection

> "Components should be lean and focused on presenting data. They shouldn't fetch data, validate input, or log to the console. Delegate those tasks to services." -- Angular Official Documentation. This phase teaches you how to build the **brain behind the scenes** -- the services that do the real work while components focus on the UI.

---

## 5.1 What Are Services?

### The Problem: Fat Components

Imagine you have a `ProductListComponent` that does ALL of this:

```typescript
// BAD APPROACH -- everything stuffed into the component
@Component({ /* ... */ })
export class ProductListComponent implements OnInit {
  products: Product[] = [];
  isLoggedIn = false;

  ngOnInit() {
    // Fetching data directly in the component
    fetch('https://api.example.com/products')
      .then(res => res.json())
      .then(data => this.products = data);
  }

  // Authentication logic in a UI component??
  checkAuth(): boolean {
    const token = localStorage.getItem('token');
    if (token) {
      const decoded = JSON.parse(atob(token.split('.')[1]));
      return decoded.exp > Date.now() / 1000;
    }
    return false;
  }

  // Logging logic in a UI component??
  logAction(action: string): void {
    const timestamp = new Date().toISOString();
    console.log(`[${timestamp}] User performed: ${action}`);
    // Maybe send to a remote logging service too...
  }

  // Validation logic in a UI component??
  validateProduct(product: Product): boolean {
    return product.name.length > 0 && product.price > 0;
  }
}
```

**What's wrong with this?**

| Problem | Why It Hurts |
|---------|-------------|
| **Not reusable** | If `OrderComponent` also needs auth checking, you copy-paste the code |
| **Hard to test** | You can't test the auth logic without creating the entire component |
| **Violates Single Responsibility** | The component does UI + data + auth + logging + validation |
| **Hard to maintain** | Change the API URL? Hunt through every component that fetches data |
| **Tight coupling** | The component is tied to `localStorage`, `fetch`, `console.log` directly |

### The Solution: Services

A **service** is a plain TypeScript class that handles a specific piece of business logic. Components delegate work to services.

```
                    +-------------------+
                    |   Component       |
                    |   (UI only)       |
                    +-------------------+
                     /       |        \
                    v        v         v
          +-----------+  +---------+  +-----------+
          | DataService|  |AuthService| |LogService |
          | (API calls)|  |(login/  |  |(logging)  |
          |            |  | logout) |  |           |
          +-----------+  +---------+  +-----------+
```

### Real-World Analogy: The Workshop

Think of a **workshop** where multiple craftspeople work:

- **Components** = the craftspeople (each focused on building one thing)
- **Services** = the shared tools (drill press, table saw, measuring tape)
- **Dependency Injection** = the tool rack system that delivers the right tool to each craftsperson

Without services, each craftsperson would need their own drill, saw, and tape measure. That's expensive and wasteful. Instead, they share tools from a central location. Similarly, services are shared logic that any component can use.

### What Does a Service Look Like?

At its core, a service is just a class with the `@Injectable()` decorator:

```typescript
import { Injectable } from '@angular/core';

@Injectable({
  providedIn: 'root'  // Makes this service available app-wide
})
export class DataService {
  // Properties (state)
  private items: string[] = [];

  // Methods (behavior)
  getItems(): string[] {
    return this.items;
  }

  addItem(item: string): void {
    this.items.push(item);
  }
}
```

That's it. No decorators like `@Component`. No templates. No styles. Just a class with logic.

### Generating a Service with the CLI

```bash
# Full command
ng generate service services/data

# Shortcut
ng g s services/data
```

**What this creates:**
```
src/app/services/
├── data.service.ts        <-- The service class
└── data.service.spec.ts   <-- Unit test file
```

**The generated file looks like:**
```typescript
// data.service.ts (as generated by CLI)
import { Injectable } from '@angular/core';

@Injectable({
  providedIn: 'root'
})
export class DataService {

  constructor() { }
}
```

> **Note:** Unlike components, the CLI does NOT register services in any module. The `providedIn: 'root'` inside the decorator handles registration automatically. This is called **tree-shakable providers** -- more on this in section 5.3.

## 5.2 Creating Your First Service

Let's build a complete example: a **TaskService** that manages a list of tasks, and a component that uses it.

### Step 1: Create the Service

```bash
ng g s services/task
```

```typescript
// src/app/services/task.service.ts

import { Injectable } from '@angular/core';

// The @Injectable() decorator tells Angular:
// "This class can RECEIVE dependencies (other services injected into it)
//  and can BE a dependency (injected into components or other services)"
@Injectable({
  providedIn: 'root'
  // providedIn: 'root' means:
  //   1. Register this service with the ROOT injector
  //   2. Only ONE instance exists for the entire app (singleton)
  //   3. It's "tree-shakable" -- if no component uses it, it won't be in the final bundle
})
export class TaskService {

  // Private array -- only the service can modify this directly
  // Components must use the methods below
  private tasks: Task[] = [
    { id: 1, title: 'Learn Components', completed: true },
    { id: 2, title: 'Learn Data Binding', completed: true },
    { id: 3, title: 'Learn Services', completed: false },
  ];

  // Auto-incrementing ID for new tasks
  private nextId = 4;

  constructor() {
    console.log('TaskService created!');
    // This log will appear ONCE in the console because it's a singleton.
    // If you see it multiple times, something is wrong with your provider config.
  }

  // GET all tasks
  getTasks(): Task[] {
    // Return a copy so external code can't accidentally mutate our array
    return [...this.tasks];
  }

  // GET a single task by ID
  getTaskById(id: number): Task | undefined {
    return this.tasks.find(task => task.id === id);
  }

  // ADD a new task
  addTask(title: string): Task {
    const newTask: Task = {
      id: this.nextId++,
      title: title,
      completed: false
    };
    this.tasks.push(newTask);
    return newTask;
  }

  // REMOVE a task by ID
  removeTask(id: number): void {
    this.tasks = this.tasks.filter(task => task.id !== id);
  }

  // TOGGLE a task's completed status
  toggleTask(id: number): void {
    const task = this.tasks.find(t => t.id === id);
    if (task) {
      task.completed = !task.completed;
    }
  }

  // GET count of completed tasks
  getCompletedCount(): number {
    return this.tasks.filter(t => t.completed).length;
  }

  // GET count of pending tasks
  getPendingCount(): number {
    return this.tasks.filter(t => !t.completed).length;
  }
}

// The interface for a Task (can also be in a separate file like models/task.model.ts)
export interface Task {
  id: number;
  title: string;
  completed: boolean;
}
```

### Step 2: The `@Injectable()` Decorator -- Explained

```typescript
@Injectable({
  providedIn: 'root'
})
```

Let's break down what `@Injectable()` does:

| Aspect | Explanation |
|--------|-------------|
| **Purpose** | Marks a class as available to be injected as a dependency |
| **Required?** | Yes, for any service that has its own dependencies (other services injected into it). Best practice: always add it. |
| **`providedIn: 'root'`** | Registers the service with the application's root injector |
| **Why 'root'?** | Creates a single, shared instance (singleton) accessible everywhere |
| **Tree-shakable** | If no component imports this service, Angular's build process removes it from the bundle |

**What if you forget `@Injectable()`?**

```typescript
// Missing @Injectable -- this will FAIL if it has dependencies
export class BadService {
  // This service needs LoggerService injected
  constructor(private logger: LoggerService) { }
  //                          ^^^^^^^^^^^^^^
  // ERROR: Angular can't inject LoggerService because BadService
  // doesn't have @Injectable(). Angular needs the decorator to
  // generate the metadata that tells the injector what to inject.
}
```

> **Rule of thumb:** Always add `@Injectable({ providedIn: 'root' })` to every service. It's the Angular-recommended approach and it never hurts to have it even if the service has no dependencies of its own.

### Step 3: Inject the Service into a Component

```bash
ng g c task-list
```

```typescript
// src/app/task-list/task-list.component.ts

import { Component, OnInit } from '@angular/core';
import { TaskService, Task } from '../services/task.service';

@Component({
  selector: 'app-task-list',
  templateUrl: './task-list.component.html',
  styleUrls: ['./task-list.component.css']
})
export class TaskListComponent implements OnInit {

  tasks: Task[] = [];
  newTaskTitle = '';

  // DEPENDENCY INJECTION happens here in the constructor
  // Angular sees "taskService: TaskService" and:
  //   1. Looks up TaskService in the injector
  //   2. Creates an instance (or reuses existing singleton)
  //   3. Passes it as the constructor argument
  //
  // The "private" keyword is a TypeScript shortcut that:
  //   - Creates a private property called "taskService"
  //   - Assigns the injected instance to it
  //   - Makes it accessible as "this.taskService" throughout the class
  constructor(private taskService: TaskService) { }

  ngOnInit(): void {
    // Load tasks when the component initializes
    this.loadTasks();
  }

  loadTasks(): void {
    this.tasks = this.taskService.getTasks();
  }

  addTask(): void {
    if (this.newTaskTitle.trim()) {
      this.taskService.addTask(this.newTaskTitle.trim());
      this.newTaskTitle = '';     // Clear the input
      this.loadTasks();           // Refresh the list
    }
  }

  removeTask(id: number): void {
    this.taskService.removeTask(id);
    this.loadTasks();
  }

  toggleTask(id: number): void {
    this.taskService.toggleTask(id);
    this.loadTasks();
  }

  get completedCount(): number {
    return this.taskService.getCompletedCount();
  }

  get pendingCount(): number {
    return this.taskService.getPendingCount();
  }
}
```

## 5.3 Dependency Injection (DI) -- Deep Dive

### What IS Dependency Injection?

Dependency Injection is a **design pattern** (not an Angular invention). It exists in Java (Spring), C# (.NET), Python, and many other languages. The core idea is simple:

> **Instead of a class creating its own dependencies, they are provided ("injected") from the outside.**

### The Problem DI Solves

**Without DI (Tight Coupling):**

```typescript
// WITHOUT Dependency Injection -- BAD
// The component creates its own dependencies directly

class ProductListComponent {
  private dataService: DataService;
  private logger: LoggerService;

  constructor() {
    // The component CREATES its own dependencies
    this.dataService = new DataService();        // Tightly coupled!
    this.logger = new LoggerService();           // Tightly coupled!
  }

  getProducts() {
    this.logger.log('Fetching products...');
    return this.dataService.fetchProducts();
  }
}
```

**What's wrong with `new DataService()` inside the component?**

| Problem | Explanation |
|---------|-------------|
| **Can't swap implementations** | What if you want a `MockDataService` for testing? You'd have to change the component code. |
| **Can't share instances** | Every component creates its own `DataService`. If `DataService` holds state (like a cache), each component has a different cache. |
| **Hidden dependencies** | Reading the class, you don't immediately see what it needs from the outside. |
| **Hard to test** | You can't replace `DataService` with a fake in unit tests without modifying the class. |
| **Chain reaction** | If `DataService` constructor changes (e.g., now needs a URL parameter), you must update EVERY class that does `new DataService()`. |

**With DI (Loose Coupling):**

```typescript
// WITH Dependency Injection -- GOOD
// Dependencies are provided from the outside

class ProductListComponent {

  // Dependencies are DECLARED, not CREATED
  // Angular's injector provides them automatically
  constructor(
    private dataService: DataService,
    private logger: LoggerService
  ) { }

  getProducts() {
    this.logger.log('Fetching products...');
    return this.dataService.fetchProducts();
  }
}
```

**Why is this better?**

| Benefit | Explanation |
|---------|-------------|
| **Swappable** | For testing, Angular can inject `MockDataService` instead of `DataService`. No code changes needed. |
| **Shared instances** | Angular provides the same `DataService` singleton to every component that asks for it. |
| **Explicit dependencies** | Look at the constructor and you instantly know what this component needs. |
| **Testable** | In unit tests: `new ProductListComponent(mockDataService, mockLogger)` |
| **Decoupled** | The component doesn't know or care HOW `DataService` is created or configured. |

### How Angular's Injector Works (Step by Step)

When Angular encounters a component that needs services, here's what happens:

```
Step 1: Angular reads the component's constructor
        constructor(private taskService: TaskService)
                                         ^^^^^^^^^^^
        "This component needs a TaskService"

Step 2: Angular checks: "Do I already have an instance of TaskService?"
        - Looks in the component's own injector first
        - Then the parent component's injector
        - Then the module injector
        - Finally the root injector

Step 3: If found: return the existing instance (singleton behavior)
        If NOT found: check the provider configuration to learn HOW to create one

Step 4: If creating new: call "new TaskService()" and store it for future requests

Step 5: Pass the instance to the component's constructor

Step 6: Component is ready to use the service
```

**Visualizing the process:**

```
Component Constructor
  |
  |  "I need TaskService"
  v
Angular's Injector
  |
  |  Is TaskService registered?
  |  +-- YES: Does an instance exist?
  |  |     +-- YES --> Return existing instance
  |  |     +-- NO  --> Create new instance, store it, return it
  |  +-- NO: ERROR! "No provider for TaskService"
  v
Component receives the service instance
```

### The Injector Hierarchy

Angular doesn't have just one injector. It has a **tree of injectors** that mirrors your component tree:

```
Root Injector (Application level)
  |
  |-- AppModule Injector
  |     |
  |     |-- FeatureModule Injector (e.g., AdminModule)
  |     |     |
  |     |     |-- AdminComponent Injector
  |     |     |     |
  |     |     |     |-- AdminDashboardComponent Injector
  |     |     |     |-- AdminUsersComponent Injector
  |     |
  |     |-- AppComponent Injector
  |           |
  |           |-- HeaderComponent Injector
  |           |-- MainComponent Injector
  |           |     |
  |           |     |-- ProductListComponent Injector
  |           |     |-- ProductDetailComponent Injector
  |           |
  |           |-- FooterComponent Injector
```

**How the hierarchy resolves dependencies:**

When `ProductListComponent` asks for `TaskService`:
1. Check `ProductListComponent`'s own injector -- not found
2. Check `MainComponent`'s injector -- not found
3. Check `AppComponent`'s injector -- not found
4. Check `AppModule`'s injector -- not found
5. Check `Root Injector` -- FOUND! (because `providedIn: 'root'`)

This "walk up the tree" behavior is called **hierarchical injection**.

### Three Ways to Register a Service

There are three places you can register (provide) a service, and **each one has different behavior**:

#### Option 1: `providedIn: 'root'` (Recommended)

```typescript
@Injectable({
  providedIn: 'root'   // <-- Registration happens HERE, inside the service
})
export class TaskService { }
```

| Aspect | Detail |
|--------|--------|
| **Where registered** | Inside the service's own decorator |
| **Scope** | Application-wide singleton |
| **Instance count** | ONE instance for the entire app |
| **Tree-shakable?** | YES -- if no component uses it, it's removed from the bundle |
| **When to use** | Most services. This is the default and recommended approach. |

#### Option 2: `providers` Array in a Module

```typescript
// task.service.ts -- notice: NO providedIn
@Injectable()
export class TaskService { }

// app.module.ts
@NgModule({
  providers: [TaskService]  // <-- Registration happens HERE, in the module
})
export class AppModule { }
```

| Aspect | Detail |
|--------|--------|
| **Where registered** | In the module's `providers` array |
| **Scope** | All components declared in or imported by this module |
| **Instance count** | ONE instance shared by all components in this module (and child modules for eagerly loaded modules) |
| **Tree-shakable?** | NO -- the service is always included in the bundle |
| **When to use** | When you need to provide a different implementation for different modules, or for older Angular code |

> **Important caveat with lazy-loaded modules:** If you provide a service in a **lazy-loaded module's** providers array, that module gets its own instance -- NOT the root singleton. This can cause subtle bugs where different parts of your app use different instances.

#### Option 3: `providers` Array in a Component

```typescript
// task.service.ts -- notice: NO providedIn
@Injectable()
export class TaskService { }

// task-list.component.ts
@Component({
  selector: 'app-task-list',
  templateUrl: './task-list.component.html',
  providers: [TaskService]  // <-- Registration happens HERE, in the component
})
export class TaskListComponent { }
```

| Aspect | Detail |
|--------|--------|
| **Where registered** | In the component's `providers` array |
| **Scope** | This component and its children ONLY |
| **Instance count** | A NEW instance for EACH instance of this component |
| **Tree-shakable?** | NO |
| **When to use** | When you need each component instance to have its own isolated service state |

### Comparison Table: Where to Provide

| Feature | `providedIn: 'root'` | Module `providers` | Component `providers` |
|---------|----------------------|--------------------|-----------------------|
| Singleton? | Yes (app-wide) | Yes (per module) | No (per component) |
| Tree-shakable? | Yes | No | No |
| Registration location | In the service | In the module | In the component |
| Recommended? | Yes (default) | Sometimes | Rarely |
| Use case | Most services | Module-specific services | Isolated state per component |

### When to Use Each Approach

```
Should I use providedIn: 'root'?
  |
  YES for 90% of services:
  - Shared data services (API calls)
  - Auth service
  - Logger service
  - Notification service
  - Any service where you want ONE instance app-wide

Should I use module providers?
  |
  YES when:
  - Different modules need different implementations of the same service
  - You're working with a legacy codebase that uses this pattern
  - You need a service only available within a specific feature module

Should I use component providers?
  |
  YES when:
  - Each component instance needs its own isolated state
  - A form component where each instance manages its own form state
  - A widget component where each instance tracks its own data
```

## 5.4 Service Scope and Singleton Pattern

### What is a Singleton?

A **singleton** means there is exactly ONE instance of the service, shared across the entire application. Every component that injects the service gets the SAME object.

```
                     +------------------+
                     | TaskService      |
                     | (ONE instance)   |
                     | tasks: [...]     |
                     +------------------+
                      /        |        \
                     /         |         \
            +--------+   +--------+   +--------+
            | Comp A |   | Comp B |   | Comp C |
            | sees   |   | sees   |   | sees   |
            | same   |   | same   |   | same   |
            | tasks  |   | tasks  |   | tasks  |
            +--------+   +--------+   +--------+
```

**When Component A adds a task, Components B and C see it immediately** (after calling `getTasks()` again) because they all share the same `TaskService` instance.

### Code Example: Shared State Between Components

Let's prove this with a counter service shared between two components:

**The shared service:**

```typescript
// src/app/services/counter.service.ts

import { Injectable } from '@angular/core';

@Injectable({
  providedIn: 'root'  // Singleton -- one instance for the whole app
})
export class CounterService {

  private count = 0;

  getCount(): number {
    return this.count;
  }

  increment(): void {
    this.count++;
  }

  decrement(): void {
    this.count--;
  }

  reset(): void {
    this.count = 0;
  }
}
```

**Component A (incrementer):**

```typescript
// src/app/counter-incrementer/counter-incrementer.component.ts

import { Component } from '@angular/core';
import { CounterService } from '../services/counter.service';

@Component({
  selector: 'app-counter-incrementer',
  template: `
    <div class="box">
      <h3>Incrementer Component</h3>
      <p>Current Count: {{ counterService.getCount() }}</p>
      <button (click)="counterService.increment()">+ Increment</button>
    </div>
  `
  // NO providers array -- uses the root singleton
})
export class CounterIncrementerComponent {
  constructor(public counterService: CounterService) { }
}
```

**Component B (displayer):**

```typescript
// src/app/counter-displayer/counter-displayer.component.ts

import { Component } from '@angular/core';
import { CounterService } from '../services/counter.service';

@Component({
  selector: 'app-counter-displayer',
  template: `
    <div class="box">
      <h3>Displayer Component</h3>
      <p>Current Count: {{ counterService.getCount() }}</p>
      <button (click)="counterService.reset()">Reset</button>
    </div>
  `
  // NO providers array -- uses the SAME root singleton
})
export class CounterDisplayerComponent {
  constructor(public counterService: CounterService) { }
}
```

**Parent template (uses both):**

```html
<!-- app.component.html -->
<h1>Singleton Demo</h1>
<app-counter-incrementer></app-counter-incrementer>
<app-counter-displayer></app-counter-displayer>
```

**Result:** When you click "+ Increment" in the Incrementer component, the count updates in the Displayer component too. They share the SAME `CounterService` instance.

### Code Example: Isolated Instances (Component-Level Provider)

Now, what if you WANT each component to have its **own separate counter**? Use `providers` in the component:

```typescript
// src/app/isolated-counter/isolated-counter.component.ts

import { Component } from '@angular/core';
import { CounterService } from '../services/counter.service';

@Component({
  selector: 'app-isolated-counter',
  template: `
    <div class="box">
      <h3>Isolated Counter</h3>
      <p>My Count: {{ counterService.getCount() }}</p>
      <button (click)="counterService.increment()">+ Increment</button>
      <button (click)="counterService.reset()">Reset</button>
    </div>
  `,
  providers: [CounterService]  // <-- EACH instance of this component gets its OWN CounterService
})
export class IsolatedCounterComponent {
  constructor(public counterService: CounterService) { }
}
```

**Use it multiple times:**

```html
<!-- app.component.html -->
<h1>Isolated Instances Demo</h1>
<app-isolated-counter></app-isolated-counter>   <!-- Has its own CounterService (count=0) -->
<app-isolated-counter></app-isolated-counter>   <!-- Has its own CounterService (count=0) -->
```

## 5.5 Services Calling Other Services

Services don't exist in isolation. In real applications, services often depend on other services. This creates a **dependency chain**.

### Injecting One Service into Another

The same DI mechanism that injects services into components works for injecting services into services:

```typescript
// src/app/services/logger.service.ts

import { Injectable } from '@angular/core';

@Injectable({
  providedIn: 'root'
})
export class LoggerService {

  log(message: string): void {
    const timestamp = new Date().toISOString();
    console.log(`[LOG ${timestamp}] ${message}`);
  }

  warn(message: string): void {
    const timestamp = new Date().toISOString();
    console.warn(`[WARN ${timestamp}] ${message}`);
  }

  error(message: string): void {
    const timestamp = new Date().toISOString();
    console.error(`[ERROR ${timestamp}] ${message}`);
  }
}
```

```typescript
// src/app/services/auth.service.ts

import { Injectable } from '@angular/core';
import { LoggerService } from './logger.service';

@Injectable({
  providedIn: 'root'
})
export class AuthService {

  private isAuthenticated = false;
  private currentUser: string | null = null;

  // LoggerService is INJECTED into AuthService
  // This is service-to-service injection
  constructor(private logger: LoggerService) { }

  login(username: string, password: string): boolean {
    this.logger.log(`Login attempt for user: ${username}`);

    // In a real app, this would call an API
    if (username === 'admin' && password === 'password') {
      this.isAuthenticated = true;
      this.currentUser = username;
      this.logger.log(`User ${username} logged in successfully`);
      return true;
    }

    this.logger.warn(`Failed login attempt for user: ${username}`);
    return false;
  }

  logout(): void {
    this.logger.log(`User ${this.currentUser} logging out`);
    this.isAuthenticated = false;
    this.currentUser = null;
  }

  isLoggedIn(): boolean {
    return this.isAuthenticated;
  }

  getUser(): string | null {
    return this.currentUser;
  }
}
```

Now let's add an API service that uses both:

```typescript
// src/app/services/api.service.ts

import { Injectable } from '@angular/core';
import { LoggerService } from './logger.service';
import { AuthService } from './auth.service';

@Injectable({
  providedIn: 'root'
})
export class ApiService {

  // Multiple services injected into one service
  constructor(
    private logger: LoggerService,
    private authService: AuthService
  ) { }

  fetchData(endpoint: string): any {
    // Check authentication before making API calls
    if (!this.authService.isLoggedIn()) {
      this.logger.error(`Unauthorized API call to ${endpoint}`);
      throw new Error('User is not authenticated');
    }

    this.logger.log(`Fetching data from ${endpoint}`);

    // In a real app, this would use HttpClient
    // For now, return mock data
    return { data: 'mock response', endpoint };
  }

```

### The Dependency Chain Visualized

```
ApiService
  |-- depends on --> LoggerService
  |-- depends on --> AuthService
                       |-- depends on --> LoggerService
```

**When Angular creates ApiService, here's what happens:**

```
1. Component asks for ApiService
2. Angular: "To create ApiService, I need LoggerService and AuthService"
3. Angular: "Do I have a LoggerService instance?" YES (already created) --> reuse it
4. Angular: "Do I have an AuthService instance?" NO --> need to create it
5. Angular: "To create AuthService, I need LoggerService"
6. Angular: "Do I have a LoggerService instance?" YES --> reuse the same one
7. Angular creates AuthService (with the shared LoggerService)
8. Angular creates ApiService (with the shared LoggerService and new AuthService)
9. Component receives ApiService
```

**Key insight:** The LoggerService injected into `ApiService` and the LoggerService injected into `AuthService` are the **SAME instance** (because both are `providedIn: 'root'` singletons).

### Avoiding Circular Dependencies

One thing to watch out for: **Service A depending on Service B, which depends on Service A**. This creates a circular dependency and Angular will throw an error.

```typescript
// CIRCULAR DEPENDENCY -- THIS WILL FAIL

@Injectable({ providedIn: 'root' })
export class ServiceA {
  constructor(private serviceB: ServiceB) { }  // A needs B
}

@Injectable({ providedIn: 'root' })
export class ServiceB {
  constructor(private serviceA: ServiceA) { }  // B needs A -- CIRCULAR!
}

// ERROR: "Circular dependency detected: ServiceA -> ServiceB -> ServiceA"
```

**How to fix circular dependencies:**

1. **Redesign:** Usually it means your services are doing too much. Extract shared logic into a third service.
2. **Use a mediator:** Create a `ServiceC` that both `ServiceA` and `ServiceB` depend on, instead of depending on each other.

```
BEFORE (circular):          AFTER (fixed):
A <--> B                    A --> C <-- B
```

### Example 1: LoggerService -- Logging Abstraction

**Why not just use `console.log` directly?**
- You can't easily disable logging in production
- You can't redirect logs to a remote server
- You can't add consistent formatting
- You can't test whether logging occurred

```typescript
// src/app/services/logger.service.ts

import { Injectable } from '@angular/core';
import { environment } from '../../environments/environment';

// Define log levels so we can filter by severity
export enum LogLevel {
  DEBUG = 0,
  INFO = 1,
  WARN = 2,
  ERROR = 3,
  OFF = 4   // Disables all logging
}

@Injectable({
  providedIn: 'root'
})
export class LoggerService {

  // Set the minimum log level
  // In production, you might set this to WARN or ERROR
  // In development, set it to DEBUG to see everything
  private logLevel: LogLevel = environment.production
    ? LogLevel.WARN
    : LogLevel.DEBUG;

  // Optional: store logs in memory for later retrieval
  private logHistory: string[] = [];

  debug(message: string, ...data: any[]): void {
    this.logWith(LogLevel.DEBUG, message, data);
  }

  info(message: string, ...data: any[]): void {
    this.logWith(LogLevel.INFO, message, data);
  }

  warn(message: string, ...data: any[]): void {
    this.logWith(LogLevel.WARN, message, data);
  }

  error(message: string, ...data: any[]): void {
    this.logWith(LogLevel.ERROR, message, data);
  }

  // Get all stored logs (useful for error reporting)
  getHistory(): string[] {
    return [...this.logHistory];
  }

  // Clear log history
  clearHistory(): void {
    this.logHistory = [];
  }

  // Change log level at runtime (e.g., for debugging in production)
  setLogLevel(level: LogLevel): void {
    this.logLevel = level;
  }

  private logWith(level: LogLevel, message: string, data: any[]): void {
    // Skip if the message's level is below our minimum
    if (level < this.logLevel) {
      return;
    }

    const timestamp = new Date().toISOString();
    const levelName = LogLevel[level]; // Converts enum number to string
    const formattedMessage = `[${levelName} ${timestamp}] ${message}`;

    // Store in history
    this.logHistory.push(formattedMessage);

    // Keep history from growing unbounded (last 500 entries)
    if (this.logHistory.length > 500) {
      this.logHistory.shift();
    }

    // Output to console with appropriate method
    switch (level) {
      case LogLevel.DEBUG:
        console.log(formattedMessage, ...data);
        break;
      case LogLevel.INFO:
        console.info(formattedMessage, ...data);
        break;
      case LogLevel.WARN:
        console.warn(formattedMessage, ...data);
        break;
      case LogLevel.ERROR:
        console.error(formattedMessage, ...data);
        break;
    }

    // In a real app, you might also send ERROR logs to a remote service:
    // if (level >= LogLevel.ERROR) {
    //   this.sendToRemoteLogging(formattedMessage, data);
    // }
  }
}
```

**Usage in any component or service:**

```typescript

    this.logger.info('User clicked the submit button');

  }
```

### Example 2: AuthService -- Managing Authentication State

This service manages the entire authentication lifecycle: login, logout, token storage, and exposing auth state to the rest of the app.

```typescript
// src/app/services/auth.service.ts

import { Injectable } from '@angular/core';
import { BehaviorSubject, Observable } from 'rxjs';
import { LoggerService } from './logger.service';

// Define the shape of our User object
export interface User {
  id: number;
  username: string;
  email: string;
  role: 'admin' | 'user' | 'viewer';
}

// What the login API returns (in a real app)
export interface AuthResponse {
  token: string;
  user: User;
}

@Injectable({
  providedIn: 'root'
})
export class AuthService {

  // BehaviorSubject holds the current value AND emits to new subscribers
  // We use it so any component can reactively know whether the user is logged in
  //
  // Private: only this service can call .next() to change the value
  private currentUserSubject = new BehaviorSubject<User | null>(null);
  private isAuthenticatedSubject = new BehaviorSubject<boolean>(false);

  // Public: components can subscribe to these to get updates
  // We expose them as Observable (not BehaviorSubject) so consumers can only READ, not WRITE
  currentUser$: Observable<User | null> = this.currentUserSubject.asObservable();
  isAuthenticated$: Observable<boolean> = this.isAuthenticatedSubject.asObservable();

  // Token storage key
  private readonly TOKEN_KEY = 'auth_token';
  private readonly USER_KEY = 'auth_user';

  constructor(private logger: LoggerService) {
    // On service creation, check if there's a stored session
    this.loadStoredSession();
  }

  /**
   * Attempt to log in with credentials.
   * In a real app, this would make an HTTP POST to your auth API.
   */
  login(username: string, password: string): boolean {
    this.logger.info(`Login attempt for: ${username}`);

    // MOCK: In a real app, replace this with an HttpClient call
    // this.http.post<AuthResponse>('/api/auth/login', { username, password })
    if (username === 'admin' && password === 'admin123') {
      const mockUser: User = {
        id: 1,
        username: 'admin',
        email: 'admin@example.com',
        role: 'admin'
      };
      const mockToken = 'eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.mock-token';

      this.setSession(mockToken, mockUser);
      this.logger.info(`User ${username} logged in successfully`);
      return true;
    }

    this.logger.warn(`Failed login attempt for: ${username}`);
    return false;
  }

  /**
   * Log out the current user
   */
  logout(): void {
    const user = this.currentUserSubject.value;
    this.logger.info(`User ${user?.username} logging out`);

    // Clear stored data
    localStorage.removeItem(this.TOKEN_KEY);
    localStorage.removeItem(this.USER_KEY);

    // Update the subjects (this notifies all subscribers)
    this.currentUserSubject.next(null);
    this.isAuthenticatedSubject.next(false);
  }

  /**
   * Get the current auth token (for attaching to API requests)
   */
  getToken(): string | null {
    return localStorage.getItem(this.TOKEN_KEY);
  }

  /**
   * Synchronous check -- is the user currently logged in?
   */
  isLoggedIn(): boolean {
    return this.isAuthenticatedSubject.value;
  }

  /**
   * Get the current user object synchronously
   */
  getCurrentUser(): User | null {
    return this.currentUserSubject.value;
  }

  /**
   * Check if the current user has a specific role
   */
  hasRole(role: string): boolean {
    const user = this.currentUserSubject.value;
    return user !== null && user.role === role;
  }

  // ---- Private helpers ----

  private setSession(token: string, user: User): void {
    localStorage.setItem(this.TOKEN_KEY, token);
    localStorage.setItem(this.USER_KEY, JSON.stringify(user));
    this.currentUserSubject.next(user);
    this.isAuthenticatedSubject.next(true);
  }

  private loadStoredSession(): void {
    const token = localStorage.getItem(this.TOKEN_KEY);
    const userJson = localStorage.getItem(this.USER_KEY);

    if (token && userJson) {
      try {
        const user: User = JSON.parse(userJson);
        this.currentUserSubject.next(user);
        this.isAuthenticatedSubject.next(true);
        this.logger.info(`Session restored for user: ${user.username}`);
      } catch (error) {
        this.logger.error('Failed to parse stored user data');
        this.logout(); // Clear corrupted data
      }
    }
  }
}
```

**Usage in a component:**

```typescript
// login.component.ts

@Component({
  selector: 'app-login',

      <button (click)="onLogin()">Login</button>
      <p *ngIf="errorMessage" class="error">{{ errorMessage }}</p>
    </div>

    <!-- Show user info if authenticated -->
    <div *ngIf="authService.isAuthenticated$ | async">
      <h2>Welcome, {{ (authService.currentUser$ | async)?.username }}!</h2>
      <p>Role: {{ (authService.currentUser$ | async)?.role }}</p>
      <button (click)="authService.logout()">Logout</button>
    </div>
  `

  password = '';

```

**Why use `BehaviorSubject`?**

| Feature | Regular variable | BehaviorSubject |
|---------|-----------------|-----------------|
| Multiple components get updates? | No (they read a snapshot) | Yes (they subscribe and get every change) |
| Template can use `async` pipe? | No | Yes |
| Has a current value? | Yes | Yes (that's the "Behavior" part) |
| New subscribers get current value? | N/A | Yes (immediately on subscribe) |

### Example 3: NotificationService -- Toast/Alert Notifications

This service uses `BehaviorSubject` to allow any component or service to trigger notifications, and a dedicated notification component to display them.

```typescript
// src/app/services/notification.service.ts

import { Injectable } from '@angular/core';
import { BehaviorSubject, Observable } from 'rxjs';

// Define notification types
export interface Notification {
  id: number;
  message: string;
  type: 'success' | 'error' | 'warning' | 'info';
  duration: number; // milliseconds, 0 = persistent (must be dismissed manually)
}

@Injectable({
  providedIn: 'root'
})
export class NotificationService {

  private notificationsSubject = new BehaviorSubject<Notification[]>([]);
  private nextId = 1;

  // Components subscribe to this to display notifications
  notifications$: Observable<Notification[]> = this.notificationsSubject.asObservable();

  /**
   * Show a success notification (green)
   * Auto-dismisses after 3 seconds by default
   */
  success(message: string, duration = 3000): void {
    this.addNotification(message, 'success', duration);
  }

  /**
   * Show an error notification (red)
   * Does NOT auto-dismiss by default (duration = 0) so the user can read it
   */
  error(message: string, duration = 0): void {
    this.addNotification(message, 'error', duration);
  }

  /**
   * Show a warning notification (yellow/orange)
   * Auto-dismisses after 5 seconds by default
   */
  warning(message: string, duration = 5000): void {
    this.addNotification(message, 'warning', duration);
  }

  /**
   * Show an info notification (blue)
   * Auto-dismisses after 3 seconds by default
   */
  info(message: string, duration = 3000): void {
    this.addNotification(message, 'info', duration);
  }

  /**
   * Manually dismiss a notification by ID
   */
  dismiss(id: number): void {
    const current = this.notificationsSubject.value;
    this.notificationsSubject.next(
      current.filter(n => n.id !== id)
    );
  }

  /**
   * Clear all notifications
   */
  clearAll(): void {
    this.notificationsSubject.next([]);
  }

  private addNotification(
    message: string,
    type: 'success' | 'error' | 'warning' | 'info',
    duration: number
  ): void {
    const notification: Notification = {
      id: this.nextId++,
      message,
      type,
      duration
    };

    // Add to the list
    const current = this.notificationsSubject.value;
    this.notificationsSubject.next([...current, notification]);

    // Auto-dismiss after duration (if duration > 0)
    if (duration > 0) {
      setTimeout(() => {
        this.dismiss(notification.id);
      }, duration);
    }
  }
}
```

## 5.7 Summary

### Key Takeaways

| Concept | What You Learned |
|---------|-----------------|
| **Services** | Plain classes with `@Injectable()` that hold business logic, data management, and shared functionality |
| **Why services** | Components should only handle UI logic; services handle everything else (Single Responsibility Principle) |
| **`@Injectable()`** | Decorator that enables DI for the service; always include it |
| **`providedIn: 'root'`** | Creates a tree-shakable singleton available app-wide (recommended default) |
| **Dependency Injection** | Design pattern where dependencies are provided from outside, not created inside |
| **Injector hierarchy** | Root > Module > Component; Angular walks up the tree to find providers |
| **Singleton** | Default behavior: one instance shared everywhere (state is shared) |
| **Component providers** | Creates a new instance per component (state is isolated) |
| **Service-to-service** | Services can inject other services; avoid circular dependencies |
| **BehaviorSubject** | Used in services to expose reactive state that components can subscribe to |

### Mental Model

```
+-------------------------------------------------------+
|                    Your Angular App                     |
|                                                        |
|  Components          Services            Models        |
|  +---------+        +------------+      +----------+   |
|  | UI      | -----> | Business   | ---> | Data     |   |
|  | Display | <----- | Logic      | <--- | Shapes   |   |
|  | Events  |        | API calls  |      | (interfaces) |
|  +---------+        | State mgmt |      +----------+   |
|                     | Validation |                     |
|                     +------------+                     |
|                                                        |
|  Rule: Components ASK services to do work.             |
|  Rule: Services do NOT know about components.          |
|  Rule: Services CAN use other services.                |
+-------------------------------------------------------+
```

### Common Mistakes to Avoid

| Mistake | Why It's Bad | Fix |
|---------|-------------|-----|
| Putting HTTP calls in components | Not reusable, hard to test | Create a service for API calls |
| Forgetting `@Injectable()` | Service can't have its own dependencies | Always add the decorator |
| Using `new ServiceName()` instead of DI | Loses singleton behavior, not testable | Always inject via constructor |
| Providing in component when you want singleton | Multiple instances instead of one | Use `providedIn: 'root'` |
| Circular service dependencies | Runtime error | Redesign with a mediator service |
| Exposing `BehaviorSubject` directly | External code can call `.next()` | Expose as `Observable` using `.asObservable()` |
