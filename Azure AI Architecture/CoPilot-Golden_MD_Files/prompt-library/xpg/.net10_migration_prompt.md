PART 1: RUNTIME / MEMORY / GC IMPROVEMENTS (.NET 10)
=============================================================================

Focus on memory-related improvements introduced or better-exploited in .NET 10:

### Runtime / GC:
- Take advantage of .NET 10's improved GC compaction and LOH handling by reducing unnecessary allocations and large temporary objects.
- Make sure long-running services (e.g., background workers) reuse buffers and avoid per-request large allocations.
- Leverage region-based compaction for more stable memory in high-throughput APIs.

### JIT / Stack Usage:
- Encourage patterns that let the JIT keep data on the stack or in registers instead of on the heap.
- Use `struct` and small value types where appropriate (but don't overuse them), and avoid unnecessary boxing.
- Leverage .NET 10's improved escape analysis and method inlining.

### Web API Patterns:
- Prefer minimal APIs or modern ASP.NET Core patterns that reduce per-request allocations (routing, model binding, filters, logging).
- Use `System.Text.Json` efficiently (no unnecessary intermediate strings or JObjects).
- For large request/response bodies (JSON, files, binaries), use streaming and avoid loading the whole payload into memory when not required.

### Spans and Stack-Allocated Buffers:
- Use `ReadOnlySpan<T>`, `Span<T>`, and `Memory<T>` in hot paths to avoid copying data.
- Where safe and beneficial, use `stackalloc` for small fixed-size value-type buffers that live only within a method.

### Buffer Pooling and Large Objects:
- Use `ArrayPool<T>` (or other pooling) for large or frequently allocated buffers.
- Reduce allocations on the Large Object Heap (LOH) by reusing large buffers instead of creating new ones per request.

### Library / BCL Improvements:
- Where we use `ZipArchive`, compression, or similar APIs, leverage their modern streaming patterns.
- Prefer allocation-friendly APIs and patterns in `System.IO`, `System.Buffers`, and collection usage.

PART 2: DATABASE & ADO.NET PERFORMANCE (.NET 10)
=============================================================================

### Connection Management:
- Use connection pooling effectively; avoid opening/closing connections unnecessarily.
- Use `await using` for `DbConnection`, `DbCommand`, `DbDataReader` to ensure proper async disposal.
- Prefer async database operations (`ExecuteReaderAsync`, `ExecuteNonQueryAsync`, `ExecuteScalarAsync`) to free up threads.

### ADO.NET / Low-Level Data Access:
- For high-performance read paths, consider using `DbDataReader` directly with `GetFieldValue<T>()` instead of ORM when appropriate.
- Use `CommandBehavior.SequentialAccess` for large BLOB/CLOB reads to stream data instead of buffering.
- Batch multiple commands in a single round-trip where supported (e.g., SQL Server batch statements, Npgsql batching).

### Query Optimization:
- Avoid SELECT * – only retrieve columns actually needed.
- Use parameterized queries to leverage query plan caching and prevent SQL injection.
- For bulk reads, use streaming (`yield return` with `DbDataReader`) instead of materializing entire result sets.

### Bulk Operations:
- For bulk inserts/updates, use database-native bulk APIs (e.g., `SqlBulkCopy` for SQL Server, `COPY` for PostgreSQL) instead of row-by-row inserts.
- Consider table-valued parameters (TVPs) for SQL Server when passing collections to stored procedures.

### Async Streaming:
- Use `IAsyncEnumerable<T>` to stream large result sets to the client without buffering everything in memory.
- Example pattern:
  public async IAsyncEnumerable<Customer> GetAllCustomersAsync([EnumeratorCancellation] CancellationToken ct = default)
  {
      await using var conn = new SqlConnection(_connectionString);
      await conn.OpenAsync(ct);
      await using var cmd = new SqlCommand("SELECT Id, Name FROM Customers", conn);
      await using var reader = await cmd.ExecuteReaderAsync(CommandBehavior.SequentialAccess, ct);
      while (await reader.ReadAsync(ct))
      {
          yield return new Customer
          {
              Id = reader.GetInt32(0),
              Name = reader.GetString(1)
          };
      }
  }

PART 3: ENTITY FRAMEWORK CORE IMPROVEMENTS (.NET 10 / EF Core 10)
=============================================================================

### General EF Core Best Practices:
- Upgrade to EF Core 10 (or latest stable) to benefit from query compilation caching, better SQL generation, and reduced allocations.
- Use `DbContextPooling` to reuse `DbContext` instances and reduce allocation overhead:
  builder.Services.AddDbContextPool<AppDbContext>(options =>
      options.UseSqlServer(connectionString), poolSize: 128);
  ### Query Performance:
- Use `AsNoTracking()` for read-only queries to skip change tracking overhead:
  var customers = await _context.Customers.AsNoTracking().ToListAsync();
  - Use `AsNoTrackingWithIdentityResolution()` when you need consistent object references but no change tracking.
- Prefer projections (`Select`) over loading entire entities when you only need a subset of columns:
  var dtos = await _context.Orders
      .Where(o => o.Status == OrderStatus.Pending)
      .Select(o => new OrderDto { Id = o.Id, Total = o.Total })
      .ToListAsync();
  ### Compiled Queries (for hot paths):
- Use compiled queries for frequently executed queries to eliminate query compilation overhead:
  private static readonly Func<AppDbContext, int, Task<Customer?>> _getCustomerById =
      EF.CompileAsyncQuery((AppDbContext ctx, int id) =>
          ctx.Customers.FirstOrDefault(c => c.Id == id));

  public Task<Customer?> GetCustomerByIdAsync(int id) => _getCustomerById(_context, id);
  ### Async Streaming with EF Core:
- Use `AsAsyncEnumerable()` for large result sets to stream rows without buffering:sharp
  public async IAsyncEnumerable<Product> GetAllProductsAsync([EnumeratorCancellation] CancellationToken ct = default)
  {
      await foreach (var product in _context.Products.AsNoTracking().AsAsyncEnumerable().WithCancellation(ct))
      {
          yield return product;
      }
  }
  ### Batch Operations (EF Core 7+ / 10):
- Use `ExecuteUpdateAsync` and `ExecuteDeleteAsync` for bulk updates/deletes without loading entities:
  // Bulk update: mark all old orders as archived
  await _context.Orders
      .Where(o => o.CreatedAt < cutoffDate)
      .ExecuteUpdateAsync(s => s.SetProperty(o => o.IsArchived, true));

  // Bulk delete: remove soft-deleted records
  await _context.Customers
      .Where(c => c.IsDeleted)
      .ExecuteDeleteAsync();
  ### Avoiding N+1 Queries:
- Use eager loading (`Include` / `ThenInclude`) or explicit loading to avoid N+1 query patterns:
  var orders = await _context.Orders
      .Include(o => o.Customer)
      .Include(o => o.OrderItems)
          .ThenInclude(oi => oi.Product)
      .Where(o => o.Status == OrderStatus.Pending)
      .ToListAsync();
  - Use split queries for complex includes to avoid Cartesian explosion:
  var orders = await _context.Orders
      .Include(o => o.OrderItems)
      .AsSplitQuery()
      .ToListAsync();
  ### Connection Resiliency:
- Enable retry logic for transient failures:arp
  options.UseSqlServer(connectionString, sqlOptions =>
  {
      sqlOptions.EnableRetryOnFailure(
          maxRetryCount: 5,
          maxRetryDelay: TimeSpan.FromSeconds(30),
          errorNumbersToAdd: null);
  });
  ### Interceptors for Cross-Cutting Concerns:
- Use EF Core interceptors for logging, auditing, or query modification without polluting business logic:
  public class SlowQueryInterceptor : DbCommandInterceptor
  {
      private readonly ILogger<SlowQueryInterceptor> _logger;
      public SlowQueryInterceptor(ILogger<SlowQueryInterceptor> logger) => _logger = logger;

      public override ValueTask<DbDataReader> ReaderExecutedAsync(
          DbCommand command,
          CommandExecutedEventData eventData,
          DbDataReader result,
          CancellationToken ct = default)
      {
          if (eventData.Duration.TotalMilliseconds > 500)
          {
              _logger.LogWarning("Slow query ({Duration}ms): {Sql}", 
                  eventData.Duration.TotalMilliseconds, command.CommandText);
          }
          return ValueTask.FromResult(result);
      }
  }
  ### Raw SQL for Complex Queries:
- When EF-generated SQL is inefficient, use `FromSqlRaw` or `FromSqlInterpolated`:
  var topCustomers = await _context.Customers
      .FromSqlInterpolated($@"
          SELECT c.* FROM Customers c
          INNER JOIN Orders o ON c.Id = o.CustomerId
          GROUP BY c.Id, c.Name, c.Email
          HAVING SUM(o.Total) > {threshold}
          ORDER BY SUM(o.Total) DESC")
      .AsNoTracking()
      .ToListAsync();
  ### Lazy Loading – Use Sparingly:
- Avoid lazy loading in Web APIs (causes N+1 and unpredictable DB hits).
- If you must use it, be explicit and aware of the performance cost.

PART 4: CACHING & RESPONSE OPTIMIZATION
=============================================================================

### In-Memory Caching:
- Use `IMemoryCache` for frequently accessed, rarely changing data:arp
  public async Task<List<Category>> GetCategoriesAsync()
  {
      return await _cache.GetOrCreateAsync("categories", async entry =>
      {
          entry.AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(10);
          return await _context.Categories.AsNoTracking().ToListAsync();
      });
  }
  ### Distributed Caching:
- For multi-instance deployments, use `IDistributedCache` (Redis, SQL Server, etc.).

### Response Caching & Compression:
- Enable response caching for cacheable GET endpoints.
- Enable response compression (Brotli, Gzip) to reduce payload sizes.

### Output Caching (.NET 7+):
- Use output caching middleware for full response caching at the server level:
  builder.Services.AddOutputCache();
  app.UseOutputCache();

  app.MapGet("/products", async (AppDbContext db) =>
      await db.Products.AsNoTracking().ToListAsync())
      .CacheOutput(policy => policy.Expire(TimeSpan.FromMinutes(5)));
