# Azure-GoldenStorageAccount (2.0.1)

## Overview

ALWAYS RUN A TERRAFORM PLAN TO VERIFY IF A DESTROY WILL TAKE PLACE ESPECIALLY IF YOUR PIPELINE USES AUTO APPROVE ON A TERRAFORM APPLY WITHOUT FIRST VERIFING A TERRAFORM PLAN.

This Terraform module will deploy a Cigna\eviCore standard Storage Account with the following resources:

- Storage Account with CMK
  - Note: CMK is no longer optional starting with version 2.0.0
- Key Vault and Key for CMK (Customer Managed Key).  Private Endpoint can also be created on the Key Vault.
  - An existing Key Vault can also be used. (Bring your own Key Vault and Key)
  - NOTE: Key Vault resources have a max tag limit of 15. Using more than 15 tags will cause
- Ability to create multiple Blob Containers, Files, Tables & Queues
- Diagnostic settings (At least one diagnostic setting is required for Containers, Files, Queues and Tables)
- Private endpoints for services: Blob, File, Queue, Table, Web.
  Note: Private Endpoints will be required when Public Access is disabled otherwise optional.
- Support for RASS (Roles as Self Service) when using a Key Vault that uses the ***RBAC***  permission model.

Restrictions when using this module to create a Key Vault and Key for CMK

* Key Vault permission model will be RBAC
* RBAC role will only be assigned to Storage Account using managed identity using the following role:

  * Key Vault Crypto Service Encryption User
* Private Endpoint will be required if public network access is disabled  (Existing VNet and Subnet will be required)
* Diagnostic settings log destination will use the same setting from storage account diagnostic setting.
* CMK key name will be created using the following name: storage-account-cmk-key
* Service Principal deploying this module will have the following assigned RBAC roles to the key vault:

  * Key Vault Administrator
  * Key Vault Crypto Officer
* Key Auto Rotation will not be configured in this version.

CAUTION:
WHEN CHANGING A KEY VAULT WITH ANOTHER KEY VAULT IN ANY STORAGE ACCOUNT REGARDLESS HOW THE STORAGE ACCOUNT WAS CREATED (IAC OR PORTAL) YOU SHOULD FOLLOW THESE GUIDELINES.

WHEN USING THIS MODULE TO CREATE A KEY VAULT OR USING A KEY VAULT CREATED OUTSIDE OF THIS MODULE
AND DECIDE LATER TO USE A DIFFERENT KEY VAULT, YOU MUST FIRST MANUALLY CHANGE \ DISASSOCIATE THE KEY FROM THE KEY VAULT THAT IS GOING TO BE REPLACED USING THE AZURE PORTAL ***PRIOR*** TO REPLACING THE KEY VAULT ASSIGNMENT.

DELETING\DESTROYING\REPLACING A KEY VAULT PRIOR TO DISASSOCIATING THE CMK KEY FROM THE STORAGE ACCOUNT CONFIGURATION WILL LEAVE THE CMK IN AN ORPHANED STATE AS IT WILL STILL BE POINTING TO THE KEY VAULT WHERE THE CMK KEY IS LOCATED.

WHEN THE CMK IS IN AN ORPHANED STATE, YOU WILL NOT BE ABLE TO CHANGE THE KEY WHEN THE INITIAL KEY VAULT IS DELETED RESULTING IN A PERMANENT ORPHANED STATE BREAKING THE ENCRYPTION\DECRYPTION PROCESS THUS RESULTING IN AN UNUSABLE STORAGE ACCOUNT.

IN THE EVENT A KEY VAULT IS ACCIDENTLY DELETED, THERE ARE SPECIFIC STEPS THAT CAN BE TAKEN TO CORRECT THE ORPHANED STATE AS LONG AS THE KEY VAULT WAS USING THE PROPER RECOVERY MANAGMENT SETTINGS. USING THIS MODULE TO CREATE A KEY VAULT DOES USE THE APPROPRIATE RECOVERY MANAGEMENT SETTINGS.

IF YOU ARE USING AN EXISTING KEY VAULT CREATED OUTSIDE OF THIS MODULE, IT IS IMPORTANT TO ENSURE THE KEY VAULT IS CONFIGURED TO USE THE PROPER RECOVERY MANAGEMENT SETTINGS (SOFT DELETE AND PURGE PROTECTION).

IN SHORT- THE KEY VAULT AND THE KEY WILL HAVE TO BE RECOVERED TO CORRECT THE ORPHANED STATE AND TO RESTORE THE STORAGE ACCOUNT BACK TO AN OPERATIONAL STATE.

When Key Vault is created outside of this module, the following permissions will be applied to the storage account.

  Key Vault Permission Model:

* Vault access policy model

  * Key Permissions: Get, Unwrap Key, Wrap Key
    Ref: https://learn.microsoft.com/en-us/azure/storage/common/customer-managed-keys-overview
* Azure role-based access control model

  * Role: Key Vault Crypto Service Encryption User
    Ref: https://learn.microsoft.com/en-us/azure/storage/common/customer-managed-keys-configure-existing-account?tabs=azure-portal

  Note: You must provide a Key Vault with an existing Key.

When turning off "shared_access_key_enabled", AD Authentication will be turned on and the following role assignment is required to add Blob Containers: "Storage Blob Data Contributor"  See Provider Block Requirements.

## Dependencies

* AzureRM provider version must be at least 3.87
  * Required for RASS (Role as Self Service) support
* A resource group
* For private endpoints a VNET with subnets and private DNS zone (if needed) must exist
* Key Vault and Key for CMK must exist if not using the built in module key vault feature
* Starting with version 2.0.0, the Storage Account Identity type MUST be System Assigned Identity.
  If you absolutely need to have a User Assigned Identity, you can use both.

Note: Module was tested on Terraform version 1.8.3

## Provider Block Requirements

The calling module's provider block should include the following feature.

Terraform uses Shared Key Authorization to provision Storage Containers, Blobs and other items - when Shared Key Access is disabled, you will need to enable the storage_use_azuread flag in the Provider block to use Azure AD for authentication, however not all Azure Storage services support Active Directory authentication.
The Files & Table Storage API's do not support authenticating via AzureAD and will continue to use a SharedKey to access the API's.  Reference: https://registry.terraform.io/providers/hashicorp/azurerm/latest/docs

   provider "azurerm" {
      features {}
      storage_use_azuread = !var.shared_access_key_enabled
    }

## New in Version 2.0.1

This version fixes a use case scenario where the following resources: Containers, FileShares, Queues & Tables could be **unintentionally destroyed.**

The terraform resource blocks used in this module for: Containers, FileShares, Queues & Tables now use a single terraform block for each of these resources. The terraform logic for creating these resources has a dependency on the public network access setting. If public network access is set to: Disabled, private endpoints are required in order to make the storage account  accessible from a pipeline or local machine to create these resources. Two terraform blocks were used with different names for creating each of these resources where one included a necessary dependancy on a timer delay resource block for the private endpoint creation to be established vs the other not having a dependency on the timer delay resource.

**Impacted Use Case Scenario (Any prior version)**

If a storage account was first created with public network access **Enabled** (with the minimum required firewall settings) and at a **later date public network access is Disabled**, this would cause the resources to be destroyed and re-created.

* Storage accounts created with **one or more** of these resources **AND public network access set to: Disabled**

  *Remediation*: To use this version (2.0.1), a simple modification to the terraform state file can be done with the guidance of CCOE team.

**Non-Impacted Use Case Scenarios when using this version (2.0.1)**

* Storage accounts created with zero or more of these resources **AND** public network access set to: **Enabled**
* Storage accounts created with **zero** of these resources **AND** public network access set to: **Disabled**
* * *Typical scenarios where containers, files, tables or queues were not needed\created by this module:*
    * Storage account created for a Azure Function App.
    * Storage account created for Logs (NSG Flow Logs etc.).
* Storage accounts created where containers, files, tables and queues were created outside of this module.
